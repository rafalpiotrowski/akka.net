# Akka.Streams trace-continuity sample renderings

Each markdown file in this directory is a real-data rendering of the `"Akka.Streams"`
ActivitySource output for a specific stream topology. Every span tree was built from
actual `Activity` objects captured during a live `GraphInterpreter` run — no synthetic
or hand-drawn data. The files are regenerated by running
`Akka.Streams.Tests.Implementation.StreamsTraceRenderingSpec` in the
`Akka.Streams.Tests` project; each test method corresponds to one file in this folder.

## Scenarios

| File | Graph | What it demonstrates |
| --- | --- | --- |
| [`linear-chain.md`](linear-chain.md) | `Source.Queue` → `Select` → `Sink.Seq` | Baseline single-producer trace continuity — one unbroken parent chain from the producer's span through every stage. |
| [`selectasync-user-span.md`](selectasync-user-span.md) | `Source.Queue` → `SelectAsync` → `Sink.Seq` (with user span inside the async lambda) | Verifies that user code executing inside a `SelectAsync` body (e.g. `SqlClient.ExecuteAsync`) creates spans parented to the `SelectAsync` stage span, not to a random root. |
| [`batchweighted-fan-in.md`](batchweighted-fan-in.md) | 3× traced producers → `Source.Queue` → `BatchWeighted` → slow `SelectAsync` → `Sink.Ignore` | True fan-in: three distinct producer traces are merged into one aggregate. The flushed downstream stage span carries the first producer's TraceId as primary parent and attaches `ActivityLink`s to the other two producer traces. |
| [`merge-two-sources.md`](merge-two-sources.md) | 2× traced `Source.Queue` → `Merge` → `Select` → `Sink.Ignore` | Pass-through fan-in: each element keeps its own producer's TraceId across the `Merge` stage. No links attached because each merged output has exactly one contributing input. |
| [`broadcast-two-branches.md`](broadcast-two-branches.md) | `Source.Queue` → `Broadcast(2)` → 2× `Select` → 2× `Sink.Ignore` | Fan-out: the producer's TraceId propagates to every downstream copy. Requires no stage-specific instrumentation. |
| [`untraced-tick-zero-spans.md`](untraced-tick-zero-spans.md) | `Source.Tick` → `Select` → `SelectAsync` → `Sink.Ignore` (no traced producer) | Cardinality guarantee: long-lived background streams that run without a traced producer emit **zero** `"Akka.Streams"` spans, so leaving the ActivitySource registered in production is safe. |

## Regenerating the renderings

```bash
cd src/core/Akka.Streams.Tests
AKKA_STREAMS_RENDER_TRACE_SAMPLES=1 dotnet test \
  --filter "FullyQualifiedName~StreamsTraceRenderingSpec" \
  -c Release -f net10.0
```

Day-to-day test runs without the `AKKA_STREAMS_RENDER_TRACE_SAMPLES=1` environment
variable will still run the scenarios and render the span trees in memory (to the
test output) but will **not** overwrite the committed markdown files. This keeps
the committed samples stable across test runs — otherwise every run would rewrite
them with fresh TraceId / SpanId hex strings and show up as noise in `git status`.

## How the renderings are produced

1. Each test method in `StreamsTraceRenderingSpec` materializes a small graph that
   matches one of the rows above.
2. A `StreamsActivityCollector` subscribes to the `"Akka.Streams"` ActivitySource and
   collects every started/stopped `Activity` via an `ActivityListener`.
3. After the graph drains, a `RenderAndSave` helper rebuilds the parent/child tree
   from the `ParentSpanId` pointers on the captured activities, formats it as an
   ASCII tree with inline TraceId / SpanId annotations, dumps the `ActivityLink` set
   for any span that carries links, and writes the result as a markdown file next
   to the spec source.
4. The files are checked into the repository as documentation artifacts. Anyone can
   regenerate them by running `dotnet test ... --filter "FullyQualifiedName~StreamsTraceRenderingSpec"`
   and inspecting the diff.

## Why this is useful

Trace shape on a distributed-stream pipeline is notoriously hard to reason about
from the code alone. These renderings answer two questions a maintainer would
otherwise have to run the framework to answer:

- **Which stages appear as spans under which topology?** The linear chain is
  self-explanatory; `Merge`, `BatchWeighted`, and `Broadcast` each produce
  non-obvious parent/child arrangements that become clear when you see the real
  captured tree.
- **How do fan-in `ActivityLink`s actually look in the output?** The
  `batchweighted-fan-in` rendering shows the exact `span id → span id` pointers
  that the framework emits, so downstream trace viewers' rendering can be
  validated against a known-good source of truth.
